# DrillGuard OS

Локальная рекомендательная система скрининга кандидатов на осложнения при бурении.

| | |
|---|---|
| **Версия** | **0.2.2** |
| **Направление** | Бурение и внутрискважинные работы |
| **Статус доказательной базы** | `synthetic_only` · `requires_field_validation` |
| **Лицензия** | Apache-2.0 |
| **Репозиторий** | [KonkovDV/drillguard-os](https://github.com/KonkovDV/drillguard-os) |

Проект относится к цифровизации процессов строительства и ремонта скважин: выявлению возможных осложнений при бурении и снижению времени первичной инженерной проверки.

DrillGuard OS анализирует временные ряды буровых параметров и формирует для инженера понятную **карточку события**: какой сигнал отклоняется, в каком режиме это происходит, какие данные подтверждают вывод и что необходимо проверить.

### Жёсткие ограничения (обязательно читать)

Система:

- **не** управляет буровой;
- **не** передаёт команды в АСУ ТП / SCADA / системы диспетчерского управления;
- **не** является системой противоаварийной защиты (ПАЗ / SIL);
- **не** заменяет инженера;
- **не** подтверждает наличие аварии, проявления, поглощения или прихвата.

Текущая версия проверена **только на синтетических данных**. Результаты demo/benchmark **не** являются полевой валидацией и **не** подтверждают промышленную точность, экономический эффект или готовность к автономной эксплуатации.

`heuristic_score` / `rule_score` / `screening_score` отражают силу срабатывания правил и **не являются вероятностью события**.

## Производственная задача

При бурении инженер одновременно работает с давлением на стояке, расходом бурового раствора, нагрузкой на крюке, крутящим моментом, скоростью проходки, глубиной, режимом насосов и текущей операцией.

Изменение этих параметров может быть связано с:

- возможным осложнением;
- сменой режима работы;
- коротким переходным процессом;
- шумом;
- пропуском или неисправностью измерительного канала.

Разрозненные или противоречивые сигналы увеличивают время ручного сопоставления и усложняют первичную проверку. DrillGuard OS **не заменяет** технологическое решение, а помогает определить, какое отклонение требует внимания и какие данные необходимо проверить.

### Пользователи решения

- инженер по бурению;
- буровой мастер;
- специалист по сопровождению строительства скважины;
- диспетчер бурения;
- специалист по буровым растворам;
- руководитель программы бурения.

## Входные данные

**Минимальный набор:**

| Канал | Назначение |
|---|---|
| время измерения | временная шкала |
| глубина | контекст скважины |
| давление на стояке | гидравлика |
| расход бурового раствора | циркуляция |
| нагрузка на крюке | механика |
| крутящий момент | механика |
| скорость проходки | процесс бурения |
| обороты насосов | режим насосов |
| текущая операция | режим операции |
| признаки качества данных | валидность каналов |

**При наличии дополнительно:** расход на выходе; объём в ёмкости; плотность и реология раствора; температура; содержание шлама; сведения о работе оборудования; суточные рапорты; действующие тревоги и комментарии инженеров.

Словарь каналов: [`docs/DATA_DICTIONARY.md`](docs/DATA_DICTIONARY.md).

## Принцип работы

```text
схема и качество → время → режим → причинная базовая линия
  → отклонения и согласованность сигналов → persistence / гистерезис
  → класс кандидата → карточка → архив → read-only shadow mode
```

Обработка выполняется последовательно:

1. Проверяются схема данных, временная шкала, единицы, пропуски и качество каналов.
2. Определяется текущий режим операции.
3. Строится **причинная** базовая линия только по данным, уже полученным к текущему моменту (без «заглядывания» в будущее).
4. Рассчитываются отклонения давления, расхода, нагрузки, крутящего момента и скорости проходки.
5. Проверяется согласованность нескольких сигналов.
6. Учитываются длительность и устойчивость отклонения (persistence), гистерезис и задержка снятия события.
7. Смена операции, короткий выброс и недостаток истории отделяются от возможного осложнения.
8. Формируется карточка с описанием сигнала, ограничениями и рекомендуемой проверкой.

Архитектура пакета (`drillguard/`):  
`schema` → `ingestion` → `timebase` → `quality` → `regimes` → `baseline` → `features` → `persistence` → `detector` → `events` / `explain` → `report` (+ `cli`, `api`, `dashboard`, `synthetic`).

### Текущие классы результатов

| Класс | Смысл |
|---|---|
| `possible_packoff` | кандидат на ухудшение очистки ствола / ограничение циркуляции |
| `possible_lost_circulation` | кандидат на поглощение (**не** подтверждение объёма поглощения) |
| `possible_influx_candidate` | сигнал, похожий на проявление (**не** диагностика проявления) |
| `torque_drag_anomaly` | аномалия крутящего момента и нагрузки |
| `sensor_quality_issue` | проблема качества измерений |
| `operation_change` | смена операции |
| `short_transient` | короткий выброс |
| `insufficient_history` | недостаток истории для базовой линии |
| `signal_conflict` | конфликт режима и сигналов |
| `normal_noise` / `none` | отклонение в пределах ожидаемого шума / нет события |

Сигнал, похожий на проявление (`possible_influx_candidate`), **не является диагностикой проявления**. Без данных об объёме в ёмкости, расходе на выходе и экспертной проверки такой результат остаётся только кандидатом. Аналогично, кандидат на поглощение **не** подтверждает фактический объём поглощённого раствора.

## Текущая версия и доказательная база (v0.2.2)

В версии **0.2.2** реализованы:

- проверка входных данных и качества каналов;
- обработка временной шкалы;
- определение режима операции;
- причинная базовая линия (в т.ч. без адаптации во время кандидата);
- устойчивое подтверждение отклонений, гистерезис и задержка снятия;
- разделение осложнений и информационных состояний;
- объяснимые карточки событий;
- локальный интерфейс только для чтения (dashboard / API screen);
- командная строка;
- синтетический генератор сценариев;
- автоматические проверки (pytest + CI);
- отчёты JSON и HTML;
- документация по ограничениям, безопасности и плану проверки.

### Зафиксированные артефакты (SSOT)

Источник истины по числам: [`artifacts/CLAIMS_MANIFEST.json`](artifacts/CLAIMS_MANIFEST.json).  
Сверка с опубликованными JSON: [`docs/EVIDENCE_SYNC.md`](docs/EVIDENCE_SYNC.md).

На текущем наборе артефактов проекта:

| Показатель | Значение |
|---|---|
| Автотесты (pytest) | **53** |
| Синтетические случаи benchmark | **50** (10 сценариев × 5 seeds) |
| Проверки устойчивости (red team) | **12** probes, из них **11** hard gates |
| Уровень заявлений | `synthetic_only` |
| Полевая валидация | **не выполнена** (`requires_field_validation`) |

Отдельно проверены:

- причинность базовой линии и отсутствие future leakage;
- отсутствие осложнений на нормальном потоке (`normal` gate);
- отсутствие перехода короткого выброса в тревогу осложнения;
- корректная обработка сигнала, похожего на проявление (`well_control_overclaim = false`);
- согласованность опубликованных артефактов с claims-манифестом.

### Метрики (важно)

**Не используйте** rate появления класса / `compat_appearance_rate` / `event_hit_rate` как главный KPI готовности.

**Первичны:**

- **Level A** — F1 / precision / recall / detection delay / FA/h для осложнений;
- **Level B** — interval-hit для `operation_change`, no-escalation для `short_transient`;
- **Level C** — поток и качество данных.

Эти результаты подтверждают **воспроизводимость программного контура** и работу заданных ограничений. Они **не** являются полевой валидацией.

## Практическая ценность

Целевой эффект является **гипотезой** и должен оцениваться относительно исходной базы заказчика.

Предполагаемые направления эффекта:

- сокращение времени ручного разбора разрозненных сигналов;
- более ранняя фиксация устойчивого отклонения, требующего проверки;
- повышение воспроизводимости первичной инженерной оценки;
- уменьшение числа несистемных разборов переходных режимов;
- отделение проблем качества измерений от возможных технологических осложнений;
- формирование единой карточки события для последующего анализа;
- локальная обработка производственных данных без управляющего воздействия.

Конкретное сокращение НПВ, снижение числа осложнений и экономия в денежном выражении **заранее не заявляются**. Показатели определяются после получения архива, экспертной разметки и согласования исходной базы с владельцем процесса.

## Предлагаемый формат сотрудничества

1. Получение обезличенного архива одной буровой установки или серии скважин.
2. Согласование состава каналов, единиц измерения, частоты и правил обработки пропусков.
3. Проверка временной шкалы и разметка операций.
4. Получение экспертной разметки нескольких эпизодов и записей из суточных рапортов.
5. Сравнение результатов с действующими тревогами и текущим порядком разбора событий.
6. Настройка порогов только на согласованном историческом периоде.
7. Проверка на отложенном временном интервале без перемешивания данных.
8. Запуск локального режима только для чтения.
9. Ведение журнала сформированных карточек.
10. Подготовка решения о продолжении, доработке или остановке работ.

### Первый этап (архивная проверка)

Первый этап начинается с **архивной проверки**, а не с подключения к управлению буровой:

- выбрать одну буровую установку или одну серию скважин;
- описать доступные каналы, единицы и частоту измерений;
- проверить пропуски, дубликаты, задержки и качество сигналов;
- разметить операции;
- определить подтверждённые и неподтверждённые эпизоды;
- сравнить систему с действующими тревогами;
- выполнить проверку на отложенном временном интервале;
- определить критерии перехода к режиму только чтения.

При положительном результате может быть запущен локальный **теневой режим** (shadow mode):

- без записи в системы управления;
- без автоматических команд;
- с журналированием карточек;
- с экспертной оценкой каждой карточки;
- с фиксацией полезных, ошибочных и неинформативных срабатываний.

План пилота: [`docs/PILOT_PLAN.md`](docs/PILOT_PLAN.md) · протокол валидации: [`docs/VALIDATION_PROTOCOL.md`](docs/VALIDATION_PROTOCOL.md).

### Предлагаемые показатели проверки

До начала работ необходимо согласовать:

- количество ложных тревог осложнений в час;
- задержку до подтверждённого события;
- полноту и точность по экспертной разметке;
- правильность определения режима операции;
- устойчивость к пропускам, шуму и дрейфу каналов;
- долю карточек, признанных полезными инженером;
- время от появления устойчивого отклонения до его первичной проверки;
- отсутствие управляющих воздействий;
- критерии остановки и перехода к следующему этапу.

### Необходимые данные от заказчика

Для архивной проверки желательно получить: давление на стояке; расход на входе; расход на выходе (если доступен); объём в ёмкости (если доступен); нагрузку на крюке; крутящий момент; скорость проходки; глубину; обороты насосов; текущую операцию; временные метки и единицы; действующие тревоги; суточные рапорты; экспертные комментарии по нескольким эпизодам; описание действующей процедуры разбора сигналов.

## Быстрый старт (разработчик / демонстратор)

```bash
pip install -e ".[dev]"
pytest -q
python -m drillguard.cli demo --scenario packoff --html artifacts/demo_report.html
python -m drillguard.cli benchmark --output artifacts/benchmark_results.json
python -m benchmark.run_redteam
```

Опционально:

```bash
# API (read-only screen)
pip install -e ".[api]"
python -m drillguard.api

# Dashboard HTML
pip install -e ".[dashboard]"
python -m drillguard.cli dashboard --csv path/to/series.csv --html artifacts/dashboard.html
```
